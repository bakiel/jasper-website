## Section 2: The Problem We Solve

**H2:** Why Most Financial Models Fail

**Content:**
```
DFI analysts review hundreds of applications annually.
They can identify a template-based model within seconds.

Common failure points:
• Circular references that crash or iterate incorrectly
• Hardcoded values instead of formula-driven calculations
• Disconnected sheets with no traceability
• Missing DFI-specific metrics (DSCR, IRR gates)
• No sensitivity analysis or stress testing
• Amateur formatting that signals inexperience

JASPER 28 addresses each of these systematically.
```

**Layer Details:**

### Layer 1: Navigation (2-3 sheets)
```
Purpose: Orientation and quick access
Contains:
• INDEX - Hyperlinked model map
• Dashboard - Key metrics at a glance
• Model documentation

Why it matters:
DFI analysts navigate models frequently. Clear structure
signals professionalism and saves their time.
```

### Layer 2: Input (3-5 sheets)
```
Purpose: Single source of truth for all assumptions
Contains:
• Assumptions - All input parameters
• Timeline - Project phases and milestones
• Units & Constants - Standardised conversions
• Cost Escalation - Inflation and price indices

Why it matters:
Every calculation traces back to this layer.
Change one assumption, entire model updates.
No hardcoded values anywhere else.
```

### Layer 3: Funding (4-6 sheets)
```
Purpose: Capital structure and deployment
Contains:
• CAPEX Schedule - Itemised capital expenditure
• Startup Costs - Pre-operational expenses
• Loan Amortisation - Debt service calculations
• Funding Structure - Debt/equity/grant mix
• Drawdown Schedule - Phased capital deployment

Why it matters:
DFIs need to see exactly how funds will be used.
Phased deployment reduces risk perception.
```

### Layer 4: Revenue (3-5 sheets)
```
Purpose: Income generation model
Contains:
• Revenue Model - Core revenue drivers
• Pricing Analysis - Unit economics
• Volume Projections - Capacity and utilisation
• Market Segments - Customer breakdown

Why it matters:
Revenue assumptions are most scrutinised.
Bottom-up builds are more credible than top-down.
```

### Layer 5: Operations (4-6 sheets)
```
Purpose: Cost structure and operational efficiency
Contains:
• OPEX Summary - Operating expense breakdown
• Personnel - Staffing and payroll
• COGS - Cost of goods sold
• Depreciation - Asset value allocation
• Working Capital - Operational cash needs

Why it matters:
Operational efficiency determines sustainability.
Detailed breakdown shows management competence.
```

### Layer 6: Financials (5-7 sheets)
```
Purpose: Integrated financial statements
Contains:
• Income Statement - P&L projection
• Cash Flow Statement - Liquidity analysis
• Balance Sheet - Financial position
• Monthly Cash Flow (Year 1) - Detailed first year
• Tax Calculations - Fiscal obligations

Why it matters:
Three-statement integration is non-negotiable.
Cash flow timing can make or break projects.
```

### Layer 7: Analysis (4-6 sheets)
```
Purpose: Investment decision support
Contains:
• Financial Ratios - DSCR, ROE, ROA
• Sensitivity Analysis - Variable stress testing
• Scenario Analysis - Multiple outcome paths
• Valuation - NPV, IRR, payback period
• Executive Summary - Investment highlights

Why it matters:
DFIs need to see downside scenarios.
Strong ratios at base case aren't enough.
```

## Section 5: Technical Standards

**H2:** Engineering Principles

### Formula Integrity
```
Every cell contains either:
1. An input (clearly marked, in Assumptions)
2. A formula (traceable to inputs)
3. Nothing

Zero hardcoded values in calculation sheets.
Zero circular references except the one permitted:
Debt → Interest → Cash Flow → Debt (solved mathematically)
```

### Traceability
```
Any output can be traced back to assumptions in <30 seconds.
Audit trail built into structure.
Formula comments where complex calculations exist.
```

### Error Handling
```
Input validation on all assumption cells.
Error checks flag inconsistencies.
Balance sheet must balance (automated check).
Cash flow reconciles to balance sheet movements.
```

### Formatting Standards
```
• Century Gothic font throughout
• Consistent colour coding:
  - Blue: Inputs
  - Black: Formulas
  - Green: Links to other sheets
  - Red: Warnings/checks
• Row heights and column widths standardised
• Print areas defined for all sheets
```
